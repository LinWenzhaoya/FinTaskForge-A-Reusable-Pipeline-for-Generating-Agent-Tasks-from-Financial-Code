# 复用性实验:两题边际成本对比与结论(skfolio_001 vs 002)

> GPT探讨13 的核心产出。目的不是"多做一道题",而是用第二题检验"可复用造题管线"是否成立。

## 一、硬约束是否守住

> 约束:构造和验收 skfolio_002 期间,**不允许修改 `pipeline/validate_case.py`**。

**守住了。** 封装 002 只新增/修改了 `cases/skfolio_002/`(metadata + task + grader + agent_run),Runner v0.2 零改动即完成验收(6/7,唯一 FAIL 是被测 Agent 不完整修复触发 anti-hack,非 Runner 缺能力)。

## 二、边际成本对比

| 环节 | skfolio_001 | skfolio_002 | 变化 |
|---|---|---|---|
| 环境恢复 | 高(踩坑:先试 empyrical 淘汰,再定 skfolio,手动 pin 同期依赖) | **~0**(直接复用 .envs/skfolio,零新增安装) | 大幅下降 |
| 候选查找 | 高(挖多个 commit + 评估) | 中(同仓库 git log 已熟,快速锁定 variance bug) | 下降 |
| base-fail/gold-pass 验证 | 中 | 低(仓库结构已熟,一个小脚本搞定) | 下降 |
| 公开测试构造 | 中 | 低(可套 001 的形式:小数据+断言) | 下降 |
| 隐藏测试构造 | 中高(要想 anti-hack 怎么设) | 中高(仍需专家想:semi_variance 是同源陷阱) | **基本不变** |
| 任务封装 | 中(第一次定目录/脚本) | 低(套模板,README/依赖直接复制) | 下降 |
| Runner 修改次数 | —(本轮才通用化) | **0** | — |
| 验收自动化 | 手工→脚本 | 一条命令 `validate_case.py cases/skfolio_002` | 已自动化 |

## 三、用两道题回答三个问题(GPT探讨13 第五步)

### 1. Runner 是否真的通用?
**是。** 判据"第二题完全不改 Runner 即可验收"已满足。002 与 001 是不同代码路径(风险度量 vs 目标函数/继承)、不同目标文件、不同测试特征,全靠 metadata 配置差异驱动,Runner 逻辑零改动。

### 2. 环境成本是否真的下降?
**是,且下降最显著。** 002 直接复用 `.envs/skfolio`,环境恢复成本≈0。这正是 SWE-bench "同仓库多 case 摊薄环境层"的实证——第一题的环境投入(定版本、pin 依赖)被第二题免费继承。

### 3. 专家成本是否仍然存在?
**仍然存在,且集中在两处不可自动化的环节:**
- **选题 + 金融必要性判断**:哪个 bug 依赖金融知识(D2)、与已有题不同路径——靠专家。
- **anti-hack 测试设计**:002 的价值恰恰来自"想到 semi_variance 是同源陷阱"。这一步是纯专家判断,且被证明极其关键——它抓住了被测 Agent 的不完整修复。这是成本对比里**唯一基本不下降**的环节,也是这套体系的核心壁垒。

## 四、超出预期的收获:第二题捕捉到了"不完整修复"失败模式

- 001:被测 Agent 完整修复,3/3 隐藏测试全过。
- 002:被测 Agent 根因定位正确(甚至做了同文件代码考古),但**只改 variance、漏改同源 semi_variance**,anti-hack 测试命中 → 判 FAIL。
- 且已用官方完整 fix 验证:gold patch 能让 3 条隐藏测试全过,证明 anti-hack 公正(非刁难)。

**意义**:两道题恰好覆盖了"完整修复"与"局部修复"两种结果,证明这套 grader 有真实区分度,能捕捉"对了一半"这类最有评测价值的信号——而这类信号只有 anti-hack 隐藏测试能抓住,公开测试会漏判。

## 五、结论

**"可复用造题管线"第一次拿到实证:** 新增同仓库 case 时,环境与执行成本显著下降(环境≈0、Runner 零改),而选题、金融必要性判断、anti-hack 设计仍需专家逐题投入。这与业界 benchmark 的规律一致——**把成本从"每题重造"转移到"可复用的环境与执行框架",但高质量题目的核心判断无法自动化。**
