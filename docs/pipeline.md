# 金融 Agent 评测:可规模化造题管线设计与一道端到端实例

> 笔试作答 · 领域造题专家
> 主线:从专业材料出发,构造一个**可规模化的造题管线**,并用一道完整跑通的题目做端到端验证。

---

## 0. 一句话概述

我没有"手工发明一道题",而是设计了一条 **SWE-bench 式的、面向金融专业代码仓库的任务挖掘管线**:从金融开源库真实的 `Issue→Fix→Test` 链条中提取"输入—答案对",经专业性与环境筛选、去泄漏封装后,交由一个不知道答案的强 Agent 盲测,再用隐藏测试自动判分。本文档给出管线设计、一道完整跑通的实例(skfolio 最大分散化 bug),以及从踩坑中提炼的、决定规模化成败的关键规则。

---

## 1. 为什么是这条路线:两种造题范式的权衡

造题有两条路,我都实际走过,最终选择了第二条,理由如下。

| | 路线一:论文/研报改造 | 路线二:GitHub Issue 挖掘(本方案) |
|---|---|---|
| 造题动作 | **发明**任务(自己选锚点、造数据、注错、推 GT) | **挖掘**任务(用真实开发史里现成的链条) |
| 标准答案来源 | 出题人自己推导 | 维护者合并的修复 + 回归测试 |
| 主要成本 | 认知/专家成本(每换一份材料几乎从零) | 工程成本(环境恢复、封装,**可脚本化、可复用**) |
| 可规模化 | 弱 | 强(同一仓库多个 case 复用环境) |

**关键洞察**(在实际尝试路线一后得到):金融题最难的不是"评测",而是"生产一条可信、有难度、可判分的数据"。路线一把评测复杂度前置到了造题的每一步,几乎无法规模化;路线二借用了开源社区已经完成的大量前置工作(真实 bug、真实修复、真实测试),把成本转移到"可标准化、可复用"的环境工程上——这正是 SWE-bench 能成为基准的原因。

> 诚实边界:路线二的基础范式与 SWE-bench 同构,不是全新方法。**本方案的原创价值不在"用 Issue 造题"这个范式本身,而在于:面向金融垂类的专业性筛选、环境可复现性筛选、防投机的隐藏测试设计,以及对"这类题到底测什么能力"的清醒界定。**

## 2. 造题管线(6 个环节)

```
金融开源仓库池
  │
  ▼ ① 候选挖掘   扫 closed issue / merged PR / fix commit,抽取 base_commit / fix_commit / 改动文件 / 测试变化
  ▼ ② 静态筛选   按"专业性 / oracle 可靠性 / 离线性 / 环境健康度 / 泄漏风险 / 难度"打分,先淘汰再深入
  ▼ ③ 环境恢复   重建 commit 同期可运行环境(本方案最大的工程瓶颈,见 §4)
  ▼ ④ 动态验收   base 触发 bug(FAIL)+ gold 修复消除(PASS)+ 原有测试不回归
  ▼ ⑤ 任务封装   去泄漏(删 .git / issue 号 / fix / reference)+ 写题面(不给根因)+ 公开测试 + 隐藏测试
  ▼ ⑥ 盲测判分   全新无答案 Agent 解题 → 隐藏测试自动判分(含防投机)
  │
  ▼ 出厂:一道带可执行 oracle 的、经难度实测的题
```

**人机边界(半自动,不追求全自动)**:①②③④可脚本化/自动化;而"这个 bug 是否真依赖金融知识、题目是否公平、难度是否合适、是否泄漏"必须专家在环。这与 SWE-bench Verified 引入人工审核 500 样本、SWE-bench Pro 设三个人工检查点的思路一致——**高质量评测数据的核心判断难以完全自动化**。

## 3. 一道完整跑通的实例:skfolio 最大分散化 bug

### 3.1 材料

- 仓库:`skfolio/skfolio`(基于 scikit-learn 的组合优化库,2025 年活跃维护)
- 对象:`MaximumDiversification`(最大分散化组合模型)及其父类 `MeanRisk`、历史修复记录、测试体系
- 属于笔试允许的"专业模型、代码或经过验证的数据"

### 3.2 bug 与标准答案

- **现象**:当所有资产历史平均收益为负时,`MaximumDiversification().fit(X)` 误抛 `ValueError: Cannot optimize for Maximum Ratio...`
- **根因**:该类继承 `MeanRisk` 的 `MAXIMIZE_RATIO` 分支,复用了一段"防呆检查"——若所有资产预期收益 ≤ 无风险利率则报错(对最大化夏普比率合理)。但最大分散化通过 `overwrite_expected_return` 把目标函数分子换成了加权波动率,**根本不以预期收益为目标**,该检查前提不成立却仍执行。
- **修复**:检查条件加 `overwrite_expected_return is None`。
- **标准答案 = 行为规范**(非 patch 逐字一致):负收益下 MaximumDiversification 能求解 + 真正的夏普模型在负收益下仍应抛错 + 正收益正常。

> **为什么这道题需要金融知识**:根因不在"代码写错了",而在"一个金融模型不恰当地复用了另一个金融模型的前提假设"。要定位它,必须理解"最大分散化的目标函数 ≠ 夏普比率的目标函数"——纯软件视角很难判断"这个检查为什么不该对这个类生效"。

### 3.3 任务包结构(双层隔离)

```
skfolio_maxdiv_task/
├── task/                    ← 给被测 Agent(已确认零答案泄漏)
│   ├── TASK.md              只描述异常现象,不给根因
│   ├── README.md            环境/运行/提交约束(禁改测试、禁吞异常)
│   ├── requirements-pinned.txt  同期依赖(顶层 pin,非严格 lock,见 §4)
│   └── repo/                修复前代码快照(去 .git / .github)
│       └── tests/test_public_reproduction.py   公开复现测试
├── grader/                  ← 出题方保留
│   ├── hidden_tests/        3 类隐藏测试
│   └── reference.patch      官方修复(参考,非逐字标准)
└── agent_run/               被测 Agent 盲测产物(prompt/trajectory/patch)
```

### 3.4 隐藏测试:防投机是精髓

公开测试只验证"负收益能求解"——但 Agent 可以粗暴删掉整个检查来蒙混。故隐藏测试设了三类:

1. **目标行为**:MaximumDiversification 负收益下能求解
2. **防投机(关键)**:真正的 `MeanRisk(MAXIMIZE_RATIO)` 在负收益下**仍应正确抛错** —— Agent 若删掉整个检查,此条变红
3. **正常回归**:正收益下 MaximumDiversification 正常

grader 自检:修复前 = 目标 fail / 防投机+回归 pass;修复后 = 3 pass。评分器区分"真修 vs 投机"的能力得到验证。

## 4. 关键发现:环境可复现性才是规模化的真正瓶颈

这是本次实践最有价值的、踩坑换来的认知——**SWE-bench 式路线的"重",不在找 bug,而在重建 commit 同期的可运行环境。** 两个对照实例:

| | empyrical #60(淘汰) | skfolio #196(采用) |
|---|---|---|
| 库年代 | 2019 | 2025 |
| bug 金融语义 | 高(线性 vs 复利年化) | 高(目标函数 + 继承) |
| 环境问题 | 旧依赖无 wheel、要编译、缺 setuptools、新 pandas 删了旧 API | 只写依赖下限,默认装了过新依赖(cvxpy API 漂移,cp.trace 由类变函数) |
| 能否解决 | ✗ 编译连锁失败 | ✓ 降级到同期版本(有 wheel) |

由此提炼两条**硬筛选规则**:
1. **环境健康度是硬门槛,不是附加项**:必须能在主流 Python 上装、不编译多个历史依赖、不改与目标 bug 无关的源码。
2. **"有 pyproject.toml"不够,真正的门槛是"能否重建 commit 同期环境"**:仓库只写依赖下限时,checkout 历史 commit 后最新依赖会破坏旧源码,必须手动 pin 同期版本;**优先带 lockfile(uv.lock/poetry.lock)的仓库**,边际成本最低。

> 诚实边界:本方案的 `requirements-pinned.txt` 只锁了关键顶层包,不是严格 lockfile(未锁传递依赖 hash),足以在本机复现,但要"换机器 100% 复现"应进一步用 uv lock / 提供 Dockerfile。

## 5. Agent 实测(笔试第 3 条)

- **被测 Agent**:一个全新的、独立会话的强 Agent(具备代码浏览、终端执行、文件修改、运行测试能力),通过新会话隔离出题阶段的全部答案信息。之所以用它:它是当前可稳定调用、且具备完成本任务所需全部能力(读代码/跑测试/改文件)的 Agent。
- **过程**:自主完成 复现 → 读继承链 → 理解目标函数差异 → 定位根因 → 最小修复 → 自我验证,与官方修复思路一致,且**自主推理得出**(prompt 禁止联网、task/ 无 .git)。它甚至自发做了防投机自查(找出原仓库测试确认夏普保护仍生效)。
- **判分**:出题方用 grader 隐藏测试独立裁决 → **3/3 通过,判定真修(非投机)**。
- **诚实记录的干扰**:首轮被测 Agent 因宿主 CLAUDE.md 规则停下等确认,补充了一次"允许连续执行"的控制指令(未含任何解题线索);另 prompt 告知了 venv 位置(降低环境配置难度,但本题主测 bug 定位)。→ 教训:正式盲测环境应无用户自定义规则、干净隔离。

### 5.1 回归严谨性(独立复核,非采信 Agent 自述)

出题方在**同一环境**下,对被测 Agent 修复前(base)与修复后(agent-fix)版本,跑同一批测试(`test_maximum_diversification.py` + `test_mean_risk.py`,共 1099 条),对比失败集合:

| | 修复前 base | 修复后 agent-fix |
|---|---|---|
| 通过 | 1080 | 1080 |
| 失败 | 19 | 19 |

差集分析:
- **仅修复后新增的失败 = 空 → 零新增回归**(核心结论)
- 19 个失败在修复前后**完全相同**,全部是 `test_cardinality_constraint` 等需要 SCIP/MIP 求解器的用例,与本次修复的 `MAXIMIZE_RATIO` 分支无关,属环境未装可选求解器的预存问题(修复前也失败)。
- 目标复现测试 `test_public_reproduction.py`:修复前 FAIL、修复后 PASS(见 grader 判分)。

**结论**:被测 Agent 修复了目标问题,且未引入任何新增回归。此结论由出题方独立复核得出,非采信被测 Agent 自述。

## 6. 难度的诚实定位

**中等难度的真实仓库 bug 修复任务**,不夸为高难度。题面已给出类名、异常文本、触发条件和公开失败测试,Agent 沿 traceback 能较快定位到 `_mean_risk.py`,再搜 `overwrite_expected_return` 即可修复。难度主要在:代码定位 + 继承链阅读 + 金融目标函数语义理解 + 最小修复,而非"从零识别这是不是 bug"。

**这套体系主要测的是**:金融专业代码仓库中的软件工程能力(仓库导航、调试、继承推理、回归意识),附带一定金融语义理解——而非纯金融研究能力。每道题应打"领域必要性"标签(D0 纯工程 ~ D3 正确实现由金融定义决定),优先入库 D2–D3,避免题库退化成"普通 SWE-bench 的金融子集"。本题为 **D2**(不懂"最大分散化目标 ≠ 预期收益"很难正确定位)。

## 7. 规模化下一步:接入 SWE-bench Harness(架构设计,未实际迁移)

本次用轻量目录结构完成了最小闭环。真正规模化时,不重复造评测轮子,而是复用 SWE-bench 成熟的执行/评分基础设施:

```
金融候选生成层(自研:挖掘/筛选/环境恢复/去泄漏/隐藏测试)
        ↓  产出标准 instance(base_commit / patch / test_patch / FAIL_TO_PASS / PASS_TO_PASS)
SWE-bench Docker Harness(复用:三层镜像 / patch 应用 / 并行 / 日志)
        ↓
FAIL_TO_PASS + PASS_TO_PASS + 结果报告
```

复用其 Docker 三层镜像(同一仓库多 case 共用环境层,边际成本下降)、patch 应用、FAIL_TO_PASS/PASS_TO_PASS 协议、gold-patch 自检。自研价值集中在上游的金融 case 生产。**不迁移核心 Harness 代码**(避免笔试时间陷入读框架)。
